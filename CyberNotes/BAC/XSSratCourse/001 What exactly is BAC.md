### [[Access Control VS Broken Access Control]]
* Developer has to successfully implement access control
* On every endpoint
* On every object + every function to said object
* This is very easy to break or forget
* Remember ~ 1000s of endpoints and functions

---
### [[RBAC vs ABAC]]
RBAC = Role based access control
* Every user gets a role
* Roles have rights assigned to them
* A right is either tied to an objection, function or even property of an object

ABAC = Access Based Access Control
* Users have properties
* Rights are tied to properties
* For example browser, location in network, geographical location (Netflix), etc.

---
###  [[Authentication vs Authorization]]
* You donot want to keep logging in for every action
* First you log in
* Then you get a token
* Token is representation of you
* Token is used for authorization after
* When you test anything BAC or access control you use said token

### How to organize your testing
* Testing is complex
* Organize in function vs object testing
* Function testing
	* Easiest of the two
	* **List all functions** (e.g. create, read, update, delete, print, export).
	* columns is all of the users
	* Cross off what each user type should be able to do
	* All user types/groups
	* If individual rights, you will think if rights that cross each other
		* For example user can not see invoices but can print documents and invoice is document,...

* Object testing
	* Single tenant testing (1 account = 1 full setup in infrastructure)
		* Simplest type of IDOR testing
			* List Resources
			* Define users who do NOT have access to all resources, only their own
			* Test all the resources on RUD (Read Update Delete)
		* Multi tenant testing
			* List all resources
			* List users from OWN company who cannot see resouces
				* Test all RUD actions
			* List users from OTHER company who can see all resources
				* Test all RUD actions
			* List users from OTHER company who can only see their OWN resources
				* Test all RUD actions

### GUID VS Sequential IDs
Sequential IDs
### What Are They?
- Sequential IDs (also called **incremental IDs** or **auto-increment IDs**) are **ordered numbers.**
- Example:
```sql
User 1: ID = 1
User 2: ID = 2
User 3: ID = 3
```
- Commonly generated by **databases automatically** (like `AUTO_INCREMENT` in MySQL).
### Advantages:
- 🗂️ Easy to read and understand.
- 🔍 Predictable structure makes searching and sorting faster.
- ⚡ Very efficient for database indexing.
### Disadvantages:
- 🔓 **Predictable = security risk** (can make apps vulnerable to IDOR attacks).
- 🐌 Can cause **hot spots in databases** (if many inserts go to the same place).
---
## GUIDs (Globally Unique Identifiers)
### What Are They?
- GUID = **128-bit unique value** (also called UUID).
- Example:   
    `3f2504e0-4f89-11d3-9a0c-0305e82c3301`
- Looks random and **hard to guess.**
- Generated using various algorithms that include:
    - Random data        
    - Timestamp        
    - Hardware addresses        
### Advantages:
- 🔐 Very hard to guess → improves **security.**
- 🌍 Can be **generated globally** (not just by the database).
- 🔄 Safer in distributed systems (when multiple servers create records).

### Disadvantages:
- 🐢 Longer and heavier than sequential IDs.    
- 🗂️ Less efficient in database indexing.    
- ❌ Hard to read and troubleshoot.    

---
## 🚩 Summary of Differences

|Feature|Sequential IDs|GUIDs|
|---|---|---|
|Format|1, 2, 3, ...|3f2504e0-4f89-11d3...|
|Predictable|Yes|No|
|Security|Low (easy to guess)|High (hard to guess)|
|Performance|Fast indexing|Slower indexing|
|Readability|Easy|Hard|
|Global Uniqueness|No|Yes|
|Best for|Small apps, private systems|Distributed, public, or sensitive systems|

---
## 🔑 Why It Matters in Security (Example)
- 🔓 If you use sequential IDs:    
    `GET /user/1000`    
    An attacker may try:   
    `GET /user/1001 GET /user/1002`    
    → Leads to **IDOR attacks.**    
- 🔐 If you use GUIDs:        
    `GET /user/3f2504e0-4f89-11d3-9a0c-0305e82c3301`    
    → Almost impossible to guess next ID.    

---
## ✅ Best Practices:
- 🔐 Use **GUIDs for public-facing IDs** (user profiles, orders, invoices).    
- 🔢 Use **sequential IDs for internal systems** where security is less critical and performance matters.    
- 🛠️ In sensitive apps, **hide or obfuscate sequential IDs** if you must use them.
